## Notes on cardinality and constraints

- **`platforms`** is a small reference table seeded by `migrations/0002_core_content_tables.sql`
  with 8 rows (`instagram`, `twitter`, `youtube`, plus `reddit`/`linkedin`/`facebook`/`tiktok`/
  `news` reserved for future scrapers — see `docs/architecture.md`). Every content/log table with a
  `platform` column has a foreign key to `platforms.name` (not `platforms.id`).
- **`authors` -> `channels`/`posts`/`comments`**: one author can own many channels, author many
  posts, and author many comments (`on delete cascade` on all three — deleting an author cascades
  to their content).
- **`posts` <-> `videos`**: one-to-one in practice for YouTube (`videos.post_id` is nullable and
  unique-in-practice per the ingestion pipeline's pairing, though not DB-enforced unique); every
  other platform has no `videos` row at all. `videos.channel_id` is required (`not null`).
- **`posts` -> `media`**: one-to-many, `on delete cascade`.
- **`posts` <-> `hashtags`**: many-to-many through the `post_hashtags` join table (composite
  primary key `(post_id, hashtag_id)`, both `on delete cascade`).
- **`posts`/`comments` -> `mentions`**: one-to-many from either parent; a `check` constraint
  (`mentions_target_check`) requires at least one of `post_id`/`comment_id` to be set.
- **`posts` -> `engagement`**: one-to-one (`post_id` is `unique not null`).
- **`comments` -> `comments`**: self-referential `parent_comment_id` (nullable, `on delete
  cascade`) models replies without a separate `replies` table.
- **`users` -> `conversations` -> `messages`**: `conversations.user_id` is nullable
  (`on delete set null`) since conversations can exist without an authenticated user in this
  version of the app; `messages.conversation_id` is required (`on delete cascade`).
- **`conversations` -> `query_logs`/`assistant_logs`**: both nullable (`on delete set null`) so
  logs outlive a deleted conversation for audit purposes. `assistant_logs.message_id` similarly
  references `messages.id` with `on delete set null`.
- **`documents` -> `embeddings`**: one document can have multiple embedding rows in principle
  (one per `(source_id, source_type, model)` combination, enforced by a unique constraint), though
  in practice one document has exactly one embedding per configured model. `embeddings.document_id`
  is `not null` with `on delete cascade`.
- Every content/chat table listed above with a `deleted_at` column implements **soft delete**:
  application code always filters `deleted_at is null` (see `BaseRepository.list_all`) rather than
  issuing a real `DELETE`.
